### 2. ❌ Выявлены критические ошибки с массивами

**Проблема:** В 4 местах кода массив пассажиров (`['Иванов Иван', 'Петров Петр']`) напрямую записывается в SERVICE поле, которое ожидает строку.

**Последствия:**
- Bitrix24 может отклонить запрос или записать "Array"
- Система отслеживания изменений работает некорректно
- Ложные уведомления о изменении пассажиров
- Невозможность определить реальные изменения

**Места с ошибками:**

| Файл | Функция | Строка | Описание |
|------|---------|--------|----------|
| `botManager.php` | `driverAcceptHandle()` | 287 | Зарегистрированный водитель принимает заявку |
| `botManager.php` | `driverAcceptHandle()` | 361 | Незарегистрированный водитель принимает заявку |
| `botManager.php` | `driverAcceptHandle()` | 554 | Альтернативная ветка принятия заявки |
| `botManager.php` | `groupAcceptHandle()` | 1534 | Групповое принятие заявки |

**Правильная реализация** (уже есть в коде):
```php
// ✅ Корректная обработка в sendPrivateMessageToDriver() (строка 1479-1497)
$passengers = $deal[botManager::PASSENGERS_FIELD];
if (is_array($passengers)) {
    $passengers = implode(", ", $passengers);  // Конвертация в строку
}
botManager::PASSENGERS_FIELD_SERVICE => $passengers,  // Строка вместо массива
```

- **Номера заявок:** PT-1234, BTW999999, PT1234 (с дефисом и без)

### 2. Workflow заявок ✅
```
NEW → PREPARATION → [Отправка в чат] → Принять → PREPAYMENT_INVOICE →
→ Начать → EXECUTING → Завершить → FINAL_INVOICE
```

**Обработчики:**
- `newDealMessage()` - отправка в групповой чат водителей
- `driverAcceptHandle()` - принятие заявки водителем
- `groupAcceptHandle()` - групповое принятие
- `travelStartHandle()` - начало поездки
- `finishHandle()` - завершение поездки
- `cancelHandle()` - отмена заявки

### 3. Система напоминаний ✅
- **Cron задача:** каждые 5 минут
- **Логика:** за 60+ минут до выезда → отправка напоминания
- **Подтверждение:** кнопка "Подтвердить" водителю
- **Эскалация:** если нет ответа 15 минут → уведомление менеджеру
- **Файлы логов:** `/root/meetride/logs/reminder_scheduler.log`
